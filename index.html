AWS – Core Interview Questions
Q1 Explain the difference between ECS and EKS. When would you choose EKS? ▾
Medium

ECS (Elastic Container Service) is AWS-native, tightly integrated with other AWS services, and simpler to manage – great for teams not needing Kubernetes-specific features.

EKS (Elastic Kubernetes Service) is managed Kubernetes. You'd choose EKS when: (1) you need portability across clouds, (2) your team already uses Helm charts and Kubernetes operators, (3) you need advanced scheduling or custom resource definitions (CRDs).

In your resume, you've used EKS with Docker. Mention that you managed node groups, configured cluster autoscaler, and integrated with AWS IAM via IRSA (IAM Roles for Service Accounts).
Q2 How does VPC Peering differ from Transit Gateway? When did you use each? ▾
Hard

VPC Peering is a 1-to-1 connection between two VPCs. It does NOT support transitive routing — if VPC-A peers VPC-B, and VPC-B peers VPC-C, VPC-A cannot reach VPC-C automatically.

Transit Gateway is a hub-and-spoke model allowing many VPCs and on-prem networks to communicate centrally. Better for 3+ VPCs, supports route tables per attachment, and scales much better.

# Transit Gateway – hub/spoke model TGW ──── VPC-A (10.0.0.0/16) ├── VPC-B (10.1.0.0/16) ├── VPC-C (10.2.0.0/16) └── On-Prem (via VPN/Direct Connect)
Relate to your experience setting up VPCs with subnets, route tables, and internet gateways. Mention NAT Gateway for private subnet internet access.
Q3 What is the difference between Security Groups and NACLs? ▾
Easy

Security Groups are stateful – if you allow inbound traffic, the response is automatically allowed. They operate at the instance level.

NACLs (Network ACLs) are stateless – you must explicitly allow both inbound AND outbound traffic. They operate at the subnet level and support DENY rules, processed in numbered order.

Common trap: forgetting NACL is stateless. If you allow port 80 inbound, you must also allow ephemeral ports (1024-65535) outbound for responses to work.

Q5 Explain IAM Roles vs IAM Users vs IAM Policies. ▾
Medium

IAM User – Long-term identity for a human or app (has access keys). Avoid giving users direct permissions.

IAM Role – Temporary credentials assumed by services (EC2, Lambda), cross-account access, or federated users. Preferred over users for services.

IAM Policy – JSON document attached to users, roles, or groups that defines allowed/denied actions on resources (ARNs).

# Best practice flow EC2 Instance → assumes IAM Role → Role has Policy → Policy allows S3:GetObject
You implement least-privilege access and MFA. Mention using Permission Boundaries to limit what other admins can delegate.
Q6 What is CloudTrail vs CloudWatch vs Config? What do each solve? ▾
Medium

CloudTrail – Auditing & API logging. Records who did what, when, from where. Used for compliance and forensics.

CloudWatch – Metrics, logs, alarms, dashboards. Real-time operational monitoring. Used for performance and alerting.

AWS Config – Configuration compliance. Tracks resource config changes over time and checks against rules (e.g., "S3 buckets must be private").

You use CloudTrail + Config for compliance and auditing. Mention setting up Config Rules and SNS alerts for non-compliant resources.

Q2 What is the difference between a Deployment, StatefulSet, and DaemonSet? ▾
Medium

Deployment – Manages stateless pods. Supports rolling updates and rollbacks. Use for web apps, APIs.

StatefulSet – For stateful apps (databases, Kafka). Provides stable network identity and persistent storage per pod. Pods are created/deleted in order.

DaemonSet – Ensures one pod runs on every (or selected) node. Use for log collectors (Fluentd), monitoring agents (Prometheus node-exporter).

For your ELK stack, Fluentd/Filebeat runs as a DaemonSet. Elasticsearch uses StatefulSets for persistent volumes. Kibana uses a Deployment.
Q3 How does Kubernetes handle service discovery and load balancing? ▾
Medium

Kubernetes uses Services as stable endpoints for pods (which have dynamic IPs). Types:

ClusterIP → Internal only (default) NodePort → Exposes on each node's IP:port LoadBalancer→ Provisions cloud LB (AWS ELB/NLB) Ingress → HTTP/S routing, TLS termination (nginx/ALB ingress)

DNS-based service discovery: CoreDNS resolves service-name.namespace.svc.cluster.local to the ClusterIP.

Q4 What are resource limits and requests? Why are they important? ▾
Easy
resources: requests: # Minimum guaranteed resources memory: "128Mi" cpu: "250m" limits: # Maximum allowed memory: "512Mi" cpu: "500m"

Requests influence scheduling decisions. Limits enforce runtime constraints – pods exceeding memory limits are OOMKilled; CPU throttled.

Not setting limits means one runaway pod can starve all other pods on the node. Always set limits in production.
Q5 How does Horizontal Pod Autoscaling (HPA) work? ▾
Medium

HPA automatically scales pods based on observed metrics (CPU, memory, or custom). It queries the Metrics Server every 15 seconds and adjusts the replica count.

kubectl autoscale deployment my-app \ --cpu-percent=60 \ --min=2 \ --max=10
For EKS, pair HPA with Cluster Autoscaler to also scale the underlying EC2 nodes. Or use Karpenter for faster, more flexible node provisioning.
Terraform – Core Interview Questions
Q1 Explain Terraform state and why remote state matters. ▾
Hard

Terraform state is a JSON file tracking the real-world resources Terraform manages. Without it, Terraform can't know what exists or detect drift.

Remote state (S3 + DynamoDB) matters because: (1) team collaboration – everyone reads the same state, (2) locking – DynamoDB prevents concurrent applies, (3) security – state contains sensitive values.

terraform { backend "s3" { bucket = "my-tf-state-bucket" key = "prod/terraform.tfstate" region = "ap-south-1" dynamodb_table = "terraform-locks" encrypt = true } }
Q2 What is the difference between terraform plan, apply, and import? ▾
Easy
terraform plan # Dry run – shows what WILL change, no action terraform apply # Executes the plan, creates/modifies/destroys terraform import # Brings existing cloud resources under TF mgmt terraform destroy# Removes all managed resources terraform taint # Marks resource for recreation on next apply
terraform import only updates state – it doesn't write HCL config. You must write the matching .tf code manually after importing.

Q4 Explain Terraform modules and workspace usage. ▾
Medium

Modules are reusable packages of Terraform config. They encapsulate infrastructure patterns (e.g., a VPC module, an EKS module).

Q2 Explain KMS and envelope encryption. ▾
Hard

Envelope Encryption: KMS doesn't encrypt your data directly (large data is inefficient). Instead:

1. KMS generates a Data Encryption Key (DEK) 2. Your data is encrypted with DEK (locally, fast) 3. DEK itself is encrypted by KMS CMK (Customer Master Key) 4. You store encrypted data + encrypted DEK together # To decrypt: 1. Call KMS to decrypt the DEK (IAM permission needed) 2. Use decrypted DEK to decrypt your data locally
KMS key rotation can be enabled (annual) and all calls are logged in CloudTrail. Mention using KMS for S3 SSE-KMS, RDS, EBS encryption.

Real-World Scenario Questions

🔥 Scenario: Production is down. Pods are CrashLoopBackOff.

How do you debug and resolve this?

1kubectl get pods -n production — confirm which pods are crashing
2kubectl describe pod <pod-name> — check Events section for OOMKilled, liveness probe failure, image pull error
3kubectl logs <pod-name> --previous — read logs from crashed container
4Check resource limits – OOMKilled means container hit memory limit, increase limits or fix memory leak
5Check image – wrong tag? ECR permissions? Run kubectl get events --sort-by=.lastTimestamp
6Rollback if recent deploy caused it: kubectl rollout undo deployment/myapp

💸 Scenario: AWS bill doubled this month. How do you investigate?

1Go to AWS Cost Explorer → filter by service, region, account to identify the spike
2Check EC2 – unused instances, oversized types, check rightsizing recommendations
3Check data transfer costs – cross-AZ, inter-region, or NAT Gateway traffic
4Check S3 – accidental public access leading to high egress? Lifecycle policies missing?
5Set up AWS Budgets with alerts at 80% and 100% of expected monthly spend
6Enable Savings Plans or Reserved Instances for predictable workloads

🔒 Scenario: Security audit finds S3 bucket is publicly readable.

1Immediately enable S3 Block Public Access at account level and bucket level
2Audit bucket policy and ACLs, remove any public grants
3Check CloudTrail for who accessed the bucket and what data was read
4Enable AWS Config rule: s3-bucket-public-read-prohibited for ongoing compliance
5Use AWS Macie to scan for sensitive data (PII, credentials) in the bucket
6For public CDN files: use CloudFront with OAI (Origin Access Identity) instead of public S3

🚀 Scenario: How would you migrate a monolith app to Kubernetes?

1Containerize the app – write Dockerfile, ensure app is stateless (externalize sessions to Redis)
2Move secrets to AWS Secrets Manager / K8s Secrets (from env files)
3Move file storage to S3 / EFS (not local filesystem)
4Create K8s manifests: Deployment, Service, ConfigMap, HPA, PDB
5Set up Ingress with SSL termination and health checks
6Run in staging, validate, then use blue/green to cut production traffic
